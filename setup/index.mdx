Before Modeinspect can render your project, it needs to know how to **install
your dependencies and start your dev server** inside a fresh sandbox. This
page is the high-level tour of how that happens — what you do, what we do,
and what the screens look like along the way.

If you just want to get going, click **New Codebase** in the dashboard and
follow the prompts — we'll do as much as we can automatically. The pages
below are here for when something needs your attention.

## The journey at a glance

A new codebase walks through five phases. Most projects sail through the
first four without you needing to do anything.

1. **You add the codebase** — pick a GitHub repo or upload a folder.
   → [Starting a setup](./starting.md)
2. **We check it can run on Modeinspect** — your project needs to be a
   React or Next.js app with Tailwind. → [Eligibility](./starting.md#eligibility-check)
3. **We spin up a clean sandbox** — an isolated Linux VM where setup runs.
   Your local files are never touched. → [Sandbox provisioning](./starting.md#sandbox-provisioning)
4. **We try a quick auto-setup** — install dependencies and start the dev
   server. For most projects, that's all it takes. → [Minimal setup](./starting.md#minimal-setup)
5. **If we can't finish on our own, you decide what's next** — fill in the
   missing details yourself, hand it off to an AI agent, or invite a
   teammate to help. → [Auto setup](./auto-setup.md), [Manual setup](./manual-setup.md)

When the dev server comes up, your codebase is ready and you can start
building.

## Why setup exists

Locally, getting a project running is muscle memory: clone the repo, run
`pnpm install`, drop in a `.env.local`, run `pnpm dev`, open
`localhost:3000`. You don't think about it.

Modeinspect runs your project in a remote **sandbox** — a fresh Linux VM
that we spin up on demand. The sandbox has no terminal you can poke at, and
there's no persistent shell session you can leave commands running in. So we
need a recipe up front: a list of commands and files that, when replayed on
a clean machine, produce a working dev server.

That recipe is your **setup config**. Once it exists and verifies cleanly,
we can rebuild your environment in seconds, every time you (or a teammate)
opens the project.

## What setup produces

Setup config is a small JSON document with five sections. You'll see all of
these in the editor.

| Section            | What it is                                                            |
| ------------------ | --------------------------------------------------------------------- |
| **Steps**          | Ordered shell commands that build your project (`pnpm install`, etc.) |
| **Checks**         | Tests that prove each step worked (and let us skip already-done work) |
| **File steps**     | Files we upload once and replay into the sandbox (e.g. `.env.local`)  |
| **Dev server**     | The command + port that starts your app                               |
| **Env vars**       | Key/value pairs injected into every step and the dev server           |

Each has its own page in the reference section below.

## How you'll fill it in

Setup config gets filled in either by you, by our AI agent, or by both
working together.

### The fast path: you don't fill it in

For a standard JS/TS project (React or Next + Tailwind, common package
manager, no special build steps, no missing env vars), our minimal auto-
setup is enough. You click **New Codebase**, wait a couple of minutes, and
the project is ready. No editing, no decisions.

### When the fast path isn't enough

If minimal setup hits something it can't figure out — an env var your dev
server expects, a config file that's not in your repo, a setup script the
agent should run — we pause and ask what to do. You have three options:

- **Show the details and fill it in yourself** ([Manual setup](./manual-setup.md)).
  You see the editor, add the steps / files / env vars you know are
  missing, and click Verify.
- **Let the agent take over** ([Auto setup](./auto-setup.md)). An AI agent
  jumps into your sandbox, looks around, tries the obvious things, and asks
  you when it's stuck. Most projects finish here.
- **Invite a teammate** — if you're a designer and your project has setup
  steps only your developer would know, you can share a link and they can
  pick up where you left off. → [Inviting a teammate](#inviting-a-teammate)

The three options are not exclusive: you can hand it to the agent, then
edit what it produced, then verify, before saving.

## Editing setup config later

Once your codebase is set up, you don't have to re-do the whole flow if
something changes. You can edit the setup config directly from the
codebase's dashboard at any time — to add a new step, rotate an env var,
upload a new `.env.local`, change the dev server port, anything.
→ [Editing setup config](./edit-config.md)

## Inviting a teammate

You'll see an **Ask Colleague to Help** button in the setup header (and on
the New Codebase screen, before you've staged anything). It generates a
shareable link — your teammate clicks it and lands in the same setup
session you're in, with full access to the editor, the agent, and any
pending agent questions.

Useful when:

- You don't have GitHub admin and need a developer to connect the
  organization first.
- The agent is asking for a value (an API key, a database URL) that only
  your developer knows.
- You're stuck on a step and want a second pair of eyes.

The teammate needs to be in the same Modeinspect organization. The link
stops working once the session completes.
