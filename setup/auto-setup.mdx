Auto setup is what runs when our quick install-and-launch attempt
([minimal setup](./starting.md#minimal-setup)) doesn't fully bring your
dev server up on its own. There's no extra click — the moment minimal
setup hits something it can't handle, we hand the sandbox to an AI
agent automatically.

You'll see the dashboard switch from the "Running initial setup…" card
to a live view of the agent at work.

## What the agent does

An AI agent gets dropped into your sandbox with read/write access to
your codebase and a shell. It:

1. **Reads your project** — looks at `package.json`, your README, the
   structure of the repo, anything that hints at how to run things.
2. **Tries commands** — runs the install, runs your dev script, watches
   what fails, fixes things, retries.
3. **Writes config and files** as it goes — if it figures out that you
   need a `.env.local` with `DATABASE_URL=…`, it'll create that file.
4. **Records each successful step** as a setup step you can review.
5. **Asks you when it has to** — if it needs a value only you know (an
   API key, a flag for a private feature), it stops and prompts you.

You can watch this happen live. The dashboard shows the agent's running
output, the steps it has recorded so far, and any pending question.

## When the agent asks you something

Some things the agent simply can't guess — secret values, a choice
between two valid paths, files only you have. When it hits one, it pauses
and surfaces the question right in the dashboard.

You'll see one of these forms, depending on what the agent needs:

- **Env vars.** A list of `KEY=VALUE` rows for variables the agent
  detected your project reading. You fill in the values and click submit.
- **Files.** A small uploader with a suggested destination path
  (e.g. `.env.local`, `config/credentials.json`). You drop the file in
  and submit.
- **A free-form value.** A single text input for something the agent
  needs as a string.
- **A choice.** A radio list of options the agent narrowed it down to —
  pick one.

**Ask Colleague to Help** from the page header. They land on the same
session and can answer the prompt directly.

## When the agent finishes

The agent finishes by asking for a **verification run**. We discard the
agent's working sandbox, spin up a fresh one, and replay the config the
agent recorded. If everything passes — install runs clean, every step
exits OK, the dev server comes up — the setup is done.

If verification fails, the session pauses with a "Verification failed"
banner and the failing step highlighted. From there you can:

- Fix the step in the editor and re-run **Verify**
- Click **Let the agent try again** to give the agent another pass with
  the failure as context
- Hand it off to a teammate via the share link

## Why verification fails (when the agent thought it succeeded)

The verification sandbox is a fresh VM. The agent's working sandbox had
state the agent built up — installed CLIs, cached files, environment
tweaks — that don't carry over to a clean replay. If the agent didn't
record those as steps, verification won't reproduce them.

Common cases:

- A globally installed CLI the agent ran but didn't add as a step.
- A file the agent created in `/tmp` and read back, but never persisted
  as a [file step](./file-steps.md).
- An env var the agent exported in its session but didn't add to your
  setup config.

The fix is almost always one or two extra steps in the editor — open the
editor, add the missing piece, hit Verify again.

## When the agent gives up

The agent has time and step budgets. If it spends too long going in
circles — typically a sign your project has a problem only a human can
diagnose — it pauses with a summary of what it tried.

You'll see the same dashboard as a verification failure, except the
banner reads "Automated setup hit the iteration limit" or similar. From
there you can edit the config yourself, give the agent another go, or
share the session with a developer. → [Inviting a teammate](./index.md#inviting-a-teammate)

## What the agent can and can't do

It **can**:

- Run any shell command that would work in a normal Linux dev environment
- Read and write files in your project
- Start and stop processes (including your dev server)
- Ask you for clarification (env var values, files, choices)
- Add file steps for files it had to create that aren't in your repo

It **can't**:

- Access secrets you haven't given it
- Reach private package registries without credentials
- Install OS-level dependencies that need root (the sandbox is unprivileged)
- Decide for you between two valid choices — it'll either pick the more
  common option or pause and ask

## After auto setup finishes

The output is the same shape as a manual setup config — see [Manual
